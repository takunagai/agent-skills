# refund-request

返金・キャンセル請求メールを、**交渉力のある文面**で作成するスキルです。「解約を忘れてた」「商品が壊れてた」「説明と違う」——そんなときに、相手企業のポリシーを調査した上で、返金成功率の高いメールを英語・日本語で作成します。

---

## インストール

このリポジトリを clone し、スキル本体（`skills/refund-request`）を各エージェントのスキルディレクトリへ **symlink** します。

```bash
git clone git@github.com:takunagai/agent-skills.git ~/Projects/agent-skills

# ハブ（~/.agents）から実体へ絶対 symlink
ln -s /Users/$USER/Projects/agent-skills/skills/refund-request ~/.agents/skills/refund-request

# Claude Code 用に ~/.agents への相対 symlink
ln -s ../../.agents/skills/refund-request ~/.claude/skills/refund-request
```

Claude の「プロジェクト」で使う場合は、`skills/refund-request` フォルダ内の全ファイルをプロジェクト知識にアップロードしても動作します。

スキル本体の構成（人間用マニュアルである本ファイルは `docs/` に置いています）:

```
skills/refund-request/
├── SKILL.md                       # スキル本体（ワークフロー定義）
└── references/
    ├── scenarios.md               # シナリオ別の戦略・成功率の目安
    ├── optional-elements.md       # 10 個のオプション要素 + 選定フローチャート
    ├── checklist.md               # 送信前チェックリスト
    └── followup-strategy.md       # 4 段階のフォローアップ戦略
```

## できること

| 機能 | 説明 |
|------|------|
| ポリシー自動調査 | 相手企業の返金ポリシーを調査し、有利な条件と成功率の見積もりを共有 |
| メール文面作成 | 「丁寧だが毅然」と「簡潔」の 2 パターンを自動生成 |
| 英語・日本語対応 | 海外サービスへの英文メールも、国内サービスへの日本語メールも対応 |
| 送信前チェックリスト | 今回のケースに該当する項目だけを抽出して提示 |
| フォローアップ戦略 | 返信が来ない場合の段階的エスカレーション計画（具体的な日付とテンプレ付き） |
| オプション要素 | 状況に応じて文面の交渉力を高める 10 種類のオプションを提案 |

### 対応シナリオ

- サブスク自動更新の解約忘れ（一番よくあるケース）
- サービス内容が説明と異なる（広告と違う、機能が足りない等）
- 商品の不具合・欠陥（物理商品もデジタル商品も）
- その他（重複購入、未配送、不正利用、無料トライアルの意図しない課金など）

## 使い方

```
あなたの指示
  ↓
Step 1: 状況をヒアリング（不足情報を質問）
Step 2: 相手企業の返金ポリシーを自動調査 → 要約・満たす条件・成功率を共有
Step 3: オプション要素を提案 → 選択（「おまかせ」も可）
Step 4: メール文面を 2 パターン生成
Step 5: 送信前チェックリストを提示（該当項目のみ）
Step 6: フォローアップ戦略を提示（具体的な日付付き）
```

**Tips:**
- 最初のメッセージに情報をまとめて伝えると Step 1 を省略できます。
- オプション選びが面倒なら「おまかせ」と伝えれば自動選定します。

## オプション要素（交渉力を高める 10 種）

| # | オプション | 効果 | 使う場面 |
|---|-----------|------|---------|
| 1 | 自己責任の認識 | 誠実な顧客という印象 | 解約忘れ、誤購入 |
| 2 | 将来の再利用意思 | 柔軟な対応を引き出す | サービス自体は好きな場合 |
| 3 | 返金手数料の負担了承 | 返金ハードルを下げる | サブスク系全般 |
| 4 | 利用規約の引用 | 適当な対応を防ぐ | 有利な条件がある場合 |
| 5 | 代替案の受容 | 何らかの補償を引き出す | 全額返金が難しそうな場合 |
| 6 | 期限内の強調 | 対応の義務感を持たせる | 返金期限がある場合 |
| 7 | 証拠の添付 | 主張の信頼性を向上 | 不具合、内容相違 |
| 8 | 自動更新停止済み | 返金処理に集中させる | サブスク系全般 |
| 9 | 更新通知の未受領 | 通知義務不履行を指摘 | 通知が届いていない場合 |
| 10 | 顧客歴のアピール | 特別な配慮を引き出す | 数ヶ月以上の利用歴がある場合 |

## 成功率を上げるコツ

1. **早ければ早いほど有利** — 課金に気づいたらすぐ行動。7 日以内が理想。
2. **事実ベースで書く** — 感情的な文面は逆効果。冷静に、でも毅然と。
3. **自分の非は "regret" レベルで** — 「反省してます」は OK、「全部自分が悪い」は言いすぎ。
4. **ポリシーを味方にする** — 利用規約に有利な条件があれば引用する。
5. **証拠を用意する** — スクリーンショットは説得力を大幅に向上。
6. **1 通目で情報を出し切る** — やりとりの往復が少ないほど解決が速い。

### フォローアップのタイムライン

```
Day 0    初回メール送信
Day 3    リマインダー（初回メールへの「返信」で同一スレッドに）
Day 7    エスカレーション示唆（別チャネルも併用）
Day 14   最終通告（chargeback / 消費者保護機関を予告）
Day 21+  外部手段に移行（chargeback、決済プラットフォーム dispute 等）
```

---

## 注意事項

- **Chargeback は最終手段。** まず企業に直接連絡し、段階的にエスカレーションする。
- **嘘は書かない。** 「通知が来ていない」と主張する前に迷惑メールフォルダを確認するなど。
- **法的助言ではありません。** 法的手段が必要な場合は専門家に相談を。
- **不当な返金請求は非推奨。** サービスを使い倒した後の返金請求などは成功率が低いことを正直に伝えます。
