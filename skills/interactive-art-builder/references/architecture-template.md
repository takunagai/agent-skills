# 設計正本テンプレート（Phase 2）

作品の `docs/architecture.md` をこの構成で作る。**実装（委譲含む）はすべてこのファイルに従い、仕様変更は必ずここを先に更新する**（一行目にその旨を書く）。

## 必須セクションと書き方

### 1. コンセプト（concept.md の要約 3 行）

感情ゴール・コア操作・トーン。実装判断に迷ったときの最終判断基準になる。

### 2. プロセス/モジュール構成図

- ネイティブ: プロセス間の OSC 矢印図 + ポート表（例: Processing :12000 / SC :57120 / Tidal :6010）
- ウェブ: モジュール間の呼び出し関係（main → AudioEngine / visuals。契約インターフェースを明記）
- **ポート・契約の正本はこのファイル**と宣言し、実装側にはリテラルでなく定数 + 「正本は architecture.md」コメントを置かせる

### 3. 状態機械

- 正本は**入力の発生源**（Processing スケッチ / ウェブ main.ts）に一元化。音側は受けたイベントに反応するだけ（責務分離）
- 遷移図 + 各状態のパラメータ（例: `idle → charging(level 0→1, 3s easeOutQuad) → releasing(1frame) → decay(energy 1→0, 3s) → idle`）
- 短時間入力の分岐（例: 300ms 未満は軽量パス）も明記

### 4. 音視覚マッピング表（コア成果物）

1 行 1 イベントで「操作 → 視覚 → 音」の対応を固定する。委譲実装が並行しても破綻しない要。

| イベント | 引数 | 視覚反応 | 音響反応 |
|---|---|---|---|
| charge/start | x,y (0..1) | 粒子が集束開始 | ドローン起動（低音量から） |
| charge/level | level (0..1) | 収縮・輝度・ジッター ∝ level | ピッチ・LPF・音量・心拍速度が追従 |
| release | level, x, y | バースト + 衝撃波 + フラッシュ + シェイク | ドロップ + 衝撃波 + シャワー（強度 ∝ level） |
| （連続 30Hz） | amp (0..1) | グロー脈動 | ─（音 → 視覚の唯一の逆流線） |

- 音 → 視覚のフィードバックは**原則 1 本に絞る**（増やすなら本表を先に更新）

### 5. プロセス間契約

- ネイティブ: OSC アドレス・引数型・送出レート（連続値は 30Hz スロットル）の表
- ウェブ: AudioEngine インターフェース（templates/web/audio-engine-contract.ts をそのまま採用してよい。start/chargeStart 等のメソッド名は作品の動詞に合わせ改名可）

### 6. 音響設計表

SynthDef / シンセ関数ごとに 1 行: 名前・合成方式・パラメータ写像（例: `\chargeDrone ─ 鋸波 2 基デチューン + サブサイン、level→基音 40..80Hz / LPF 200..4k / 心拍 LFO 0.8..3Hz`）。マスター系（共有リバーブ + リミッタ必須）も明記。

### 7. ビジュアル設計

粒子数・描画方式（加算合成等）・パレット hex・状態別の見た目 1 行ずつ・60fps 目標と負荷調整の方針。

### 8. Tier 設計とフォールバック

- **Tier 1**: コア体験（視覚 + ワンショット音響）。単独で作品として成立すること
- **Tier 2**: パターン層（Tidal / Strudel）等のオプショナル強化。未導入・起動失敗でも Tier 1 が動くこと（クラス存在ガード / try-catch フォールバック）
- 統合テストは Tier 1 → Tier 2 の順

### 9. ディレクトリ構成・チューニング定数の置き場

定数はどのファイルの冒頭に集約するかを最初に決めて書く（README のノブ一覧表とセット）。
