# article-quality-enhancer

AI が書く記事の質を、**内容レベル**で構造的に高めるスキルです。AI に記事を書かせると、正確だけれど印象に残らない「優等生の作文」になりがちです。このスキルは 8 つの品質向上テクニックを、記事のテーマ・タイプ・読者層に応じて自動選定し、**深み・独自性・読者の引っかかり** を記事に加えます。

本スキルは **8 テクニック（#1〜#8）の定義の正本（カタログ）** です。周辺のライティング資産から次の形で参照されます:

- `content-writer` スキル（非公開）: 執筆パイプラインの「差別化パターン」表として 8 テクのダイジェストを組み込み。**定義の追加・変更は本スキルを先に更新し、content-writer 側の表を追随させる**（同期規律）
- `article-critic` / `article-editor` サブエージェント: frontmatter の `skills:` でプリロードし、採点語彙・改善提案の根拠として使用

`content-writer` との併用を明示的に指示すると、テクニック選定（本スキル）と文体・プラットフォーム最適化（content-writer）の両面から記事の品質を高められます。本スキル単独でも問題なく動作します。

> [!note] 発動は明示呼び出し専用です
> 「記事を書いて」のような一般的な執筆依頼では発動しません（執筆パイプラインを持つ他スキルとの二重発動・競合を防ぐため）。スキル名の名指し、テクニック番号・名前の指定、プロンプト提案依頼のいずれかで呼び出してください。

---

## 対応／前提

| 項目 | 内容 |
|------|------|
| 対象言語 | 日本語記事専用（テクニックの概念は言語非依存だが、具体例・表現が日本語に最適化） |
| 対応媒体 | ブログ、note、X スレッド、Instagram / Facebook 投稿など媒体を問わない |
| 発動条件 | 明示呼び出し専用（スキル名の名指し／テクニック番号・名前の指定／プロンプト提案依頼） |
| 動作モード | 執筆モード（デフォルト）／プロンプト提案モード |
| 外部依存 | なし。8 テクニックはすべて外部素材なしで単独動作する |
| 任意連携 | ユーザー固有の執筆戦略・比喩素材・体験エピソード・文体ガイドが手元にあれば、Read で引用して固有性を高められる（設置先・パスには非依存） |

## インストール

このリポジトリを clone し、スキル本体（`skills/article-quality-enhancer`）を各エージェントのスキルディレクトリへ symlink します。実体は 1 つ、参照を複数張る方式です。当環境では「実体 → `~/.agents` → `~/.claude`」の 2 段 symlink で統一しています。

```bash
git clone git@github.com:takunagai/agent-assets.git ~/Projects/agent-assets
ln -s /Users/$USER/Projects/agent-assets/skills/article-quality-enhancer ~/.agents/skills/article-quality-enhancer
ln -s ../../.agents/skills/article-quality-enhancer ~/.claude/skills/article-quality-enhancer
```

> Codex など他のエージェントは `~/.agents/skills/article-quality-enhancer` を直接読みます。Claude Code は `~/.claude/skills/` を経由して同じ実体を参照します。

スキル本体の構成は次の通りです（人間用マニュアルである本ファイルは、スキル本体の外＝`docs/` に置いています）。

```
skills/article-quality-enhancer/
├── SKILL.md                  # スキル定義（AI がトリガー時に読む指示書）
└── references/
    └── techniques.md         # 8 テクニックの詳細解説・具体例（AI 実行資産）
```

## 使い方

スキル名かテクニックを指定して呼び出します。「記事を書いて」だけでは発動しません。

### 基本：スキル名を指定して渡す（テクニック自動選定）

```
article-quality-enhancer を使って note 記事を書いて。
テーマは「30代で転職を3回した話」
```

AI がテーマと媒体から最適なテクニックを 2〜4 つ選定し、冒頭に適用テクニックを宣言してから執筆します。

### テクニックを指定して書かせる

```
以下のテーマでブログ記事を書いて。#7（逆説導入）と #3（失敗プロセス）を使って。

テーマ：プログラミング独学の始め方
読者：未経験から転職を考えている20代後半
```

番号（#1〜#8）でもテクニック名でも指定でき、指定したものが優先されます。

### リライトに使う

```
article-quality-enhancer で以下の記事の質を高めて。特に冒頭が弱いので改善したい。

（既存の記事テキスト）
```

既存記事を分析し、不足しているテクニックを適用してリライトします。「#7 を効かせて冒頭を直して」のようにテクニック指定でも呼び出せます。

### X スレッド用

```
article-quality-enhancer を使って X スレッドを作成して。

テーマ：AI ツールを使い始めて半年で仕事の進め方が完全に変わった話
ターゲット：AI に興味はあるが、まだ本格的に使っていない人
```

X スレッドに適したテクニック（逆説導入 + 数字・固有名詞）が自動選定されます。

### テクニックの組み合わせを活用する

```
以下のテーマで note 記事を書いて。#1（裏テーマ）と #5（比喩フレーム）を組み合わせて、記事全体が一つの物語になるように。

テーマ：なぜ日記を書き続けると人生が変わるのか
```

相性の良いテクニックを組み合わせ指定すると、効果が増幅します。

### プロンプト提案モード

AI に直接書かせるのではなく、他の AI ツール用のプロンプトを作りたい場合に使います。

```
「個人開発で月5万円稼ぐまでのロードマップ」というテーマで、
記事の質を高めるプロンプトを提案して
```

テクニックが組み込まれた、そのまま使えるプロンプト文面が提案されます。

## 主な機能

### 8 つのテクニック

| # | テクニック名 | ひとことで言うと | 向いている記事 |
|---|---|---|---|
| 1 | 裏テーマ | 表面テーマの奥にある真のメッセージで軸を作る | エッセイ、コラム、ビジネス |
| 2 | 対立構造 | 「A か B か」で読者に選択を迫る | 比較記事、ツール解説 |
| 3 | 失敗プロセスの注入 | 泥臭い試行錯誤で共感と信頼を作る | ハウツー、体験記 |
| 4 | 読者解像度の絞り込み | 読者の「今この瞬間」を極限まで具体化する | 技術記事、実用ガイド |
| 5 | 比喩フレーム | 記事全体を一つの比喩で貫く | 入門解説、難解なテーマ |
| 6 | 具体的な数字と固有名詞 | 曖昧表現をすべて具体情報に置換する | 全ジャンル |
| 7 | 逆説・常識の否定 | 冒頭で「え？」と思わせて離脱を防ぐ | SNS、バイラル狙い |
| 8 | 意図的な余白 | あえて答えを出さず、読者に考えさせる | エッセイ、意思決定系 |

### テクニックの自動選定

記事タイプごとに推奨テクニックのデフォルトが設定されています（あくまで初期値。実際はテーマの性質を見て柔軟に判断されます）。

| 記事タイプ | 必須級 | 推奨 |
|-----------|--------|------|
| 技術ブログ | #4 読者解像度、#6 数字・固有名詞 | #3 失敗プロセス、#7 逆説 |
| ビジネス系ブログ | #2 対立構造、#6 数字・固有名詞 | #1 裏テーマ、#7 逆説 |
| note（エッセイ） | #1 裏テーマ、#8 余白 | #5 比喩フレーム、#3 失敗プロセス |
| X スレッド | #7 逆説、#6 数字・固有名詞 | #2 対立構造 |
| SNS 投稿 | #7 逆説 | #6 数字・固有名詞、#3 失敗プロセス |
| 不明・汎用 | #1 裏テーマ、#4 読者解像度、#6 数字・固有名詞 | — |

### 適用テクニックの宣言

執筆の冒頭（記事本文の前）に、適用したテクニックを 1〜2 行で簡潔に宣言します。記事本文には「このテクニックを使っています」といったメタ的な言及は入れません。

### 執筆パイプラインスキルとの併用（明示指示時のみ）

執筆パイプラインを持つスキル（例: content-writer）と併用する場合は、「content-writer と article-quality-enhancer を併用して書いて」のように明示的に指示します。併用時の役割分担:

| 担当 | article-quality-enhancer | content-writer |
|------|------------------------|----------------|
| 内容の深み・独自性 | ○ | — |
| テクニック選定・適用 | ○ | — |
| 文体・語尾・トーン | — | ○ |
| プラットフォーム別フォーマット | — | ○ |

指示がなければ本スキルは執筆依頼に横から自動発動しません（二重発動によるワークフロー競合・トークン消費を防ぐ設計です）。片方だけでも問題なく動作します。

## 注意点

- **日本語記事専用**です。テクニックの概念自体は言語に依存しませんが、具体例や表現のニュアンスが日本語に最適化されています。
- テクニックは記事を長くするためのものではなく、内容の質を変えるためのものです。文字数を指定したい場合は「2,000文字程度で」のようにプロンプトで併記してください。
- スキル名を指定して呼び出せば、テクニックは自動選定されます。特定のテクニックを試したいときだけ番号を指定してください。
- 一部のテクニックは相性に注意が必要です。たとえば #8（意図的な余白）と #6（数字・固有名詞）は方向が逆なので、本文は #6 で密度を高め、結論だけ #8 で余白を作るなどバランスを取ります。
- ユーザー固有素材の参照は **任意** です。執筆戦略・比喩素材・体験エピソード・文体ガイドが手元にあれば引用して固有性を高められますが、なくても 8 テクニックはすべて動作します。

## カスタマイズ

- **テクニックの追加・編集**: `references/techniques.md` に新テクニックを追加するか、既存テクニックの具体例を差し替えると、自分の分野に特化できます。
- **選定ロジックの調整**: `SKILL.md` の「テクニック選定ガイド」セクションを編集すると、記事タイプごとの推奨テクニックを変更できます。
- **テクニック数の変更**: デフォルトは 2〜4 つ。`SKILL.md` の記述を変更すれば、常に 1 つだけ／5 つ以上を適用する設定にもできます。
