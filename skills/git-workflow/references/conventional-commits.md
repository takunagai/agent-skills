# Conventional Commits 詳細ガイド

## 基本構造

```
<type>(<scope>): <subject>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```

## Type 詳細

### feat（新機能）
新しい機能の追加。ユーザーに影響する変更。

```
feat(auth): ソーシャルログイン機能を追加

Google と GitHub のOAuth2.0 認証をサポート。
ユーザーは既存のアカウントとソーシャルアカウントを連携可能。
```

### fix（バグ修正）
バグの修正。ユーザーに影響する変更。

```
fix(cart): 数量が0になる問題を修正

カート内の商品数量を減らす際、0以下にならないよう
バリデーションを追加。
```

### docs（ドキュメント）
ドキュメントのみの変更。

```
docs(api): 認証エンドポイントのドキュメントを追加
```

### style（コードスタイル）
コードの意味に影響しない変更（空白、フォーマット、セミコロン等）。

```
style(components): Prettier による自動フォーマットを適用
```

### refactor（リファクタリング）
バグ修正でも機能追加でもないコード変更。

```
refactor(utils): 日付処理を dayjs に統一

moment.js から dayjs に移行。バンドルサイズを削減。
```

### perf（パフォーマンス）
パフォーマンスを向上させるコード変更。

```
perf(list): 仮想スクロールを実装

10000件以上のリスト表示時のレンダリング速度を改善。
```

### test（テスト）
テストの追加・修正。

```
test(auth): ログインフローの E2E テストを追加
```

### build（ビルド）
ビルドシステムや外部依存関係に影響する変更。

```
build(webpack): コード分割を設定

動的インポートを使用して初期ロード時間を短縮。
```

### ci（CI/CD）
CI 設定ファイルやスクリプトの変更。

```
ci(github): PR 時の自動テストを設定
```

### chore（その他）
上記に当てはまらないその他の変更。

```
chore(deps): 依存関係を更新
```

## Breaking Change

破壊的変更がある場合は、以下のいずれかの方法で示す：

### 方法1: type に `!` を付ける

```
feat(api)!: レスポンス形式を変更

以前: { data: [...] }
以後: { items: [...], meta: {...} }
```

### 方法2: footer に `BREAKING CHANGE:` を記載

```
feat(api): レスポンス形式を変更

ページネーション情報を含むよう変更。

BREAKING CHANGE: レスポンスの data フィールドが items に変更。
meta フィールドが追加され、ページネーション情報を含む。
```

## 複数の scope

関連する複数のモジュールに影響する場合：

```
feat(auth,user): プロフィール連携機能を追加
```

または scope なしで body に詳細を記載：

```
feat: プロフィール連携機能を追加

auth モジュールと user モジュールに変更。
```

## Issue/PR への参照

footer でリンク：

```
fix(login): セッションタイムアウト問題を修正

ログイン後にセッションが即座に切れる問題を解決。

Closes #123
Refs #456, #789
```

## 複数コミットを避けるべき場合

以下は1つのコミットにまとめる：
- 機能とそのテスト
- バグ修正とそのリグレッションテスト
- リファクタリングとフォーマット変更（同じファイル）

以下は分けるべき：
- 異なる機能
- 無関係なバグ修正
- ドキュメントのみの変更
