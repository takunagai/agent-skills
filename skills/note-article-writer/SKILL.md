# note 記事作成スキル

note.com 向けの記事を、段階的なワークフローで作成する。

## 書き手のクセの反映（フェーズ 0: 執筆前に必ず確認）

本スキルは設置先に依存せず単体で動作するが、**執筆系フェーズ（1〜3）を始める前に、次の 2 つを必ず確認して読み込む**。あればあるだけ「書き手の声」に近づき、なければ内蔵仕様にフォールバックする。

### 0-a. 人物像: `my-profile.md`（スキル同梱）

スキルフォルダ直下の [my-profile.md](my-profile.md) を Read する。書き手が記入していれば、次を執筆全体に反映する:

- 専門分野・価値観 → 記事の切り口と底流のメッセージ
- 想定読者 → デフォルトのターゲット（フェーズ1 の入力を補完）
- 文体の好み（一人称・使いたくない表現） → フェーズ3 の基本ルールより優先
- 書きたくないこと → 差別化パターンやタイトル案から除外

未記入（テンプレートのまま・空欄）なら読み飛ばして内蔵仕様で書く。

### 0-b. 文体サンプル: `samples/`（スキル同梱）

スキルフォルダ直下の `samples/` にある書き手の実文（`note-*.md` 等。README.md 以外）を Read し、文体を学習する:

- 文の長さの傾向とリズム（短文と長文の混ぜ方）
- 語尾のパターンと癖、一人称
- 段落の切り方、強調・比喩の使い方
- 導入と締めのパターン

**サンプルの文体は `references/style.md` の内蔵見本より優先する**（内蔵見本は samples が空のときの補助輪）。AI 生成文がサンプルに混ざっていると判断できる場合は学習に使わない。

### 0-c. 環境の追加素材（あれば）

設置先の環境に書き手のプロフィール・文体ガイド・比喩素材・体験談ストックなどのファイルがあると分かっている場合は、それも読み込んで反映する。取れない場合は 0-a / 0-b だけで問題なく動作する。

素材が多いほど「より固有性の高い記事」に、なければ「汎用テンプレートに沿った記事」になる、という関係。

## ワークフロー概要

記事作成は4つのフェーズで進める：

1. **分析** - 読者の悩みと競合差別化を明確化
2. **構造設計** - タイトル・見出し・リード文の骨格作成
3. **本文執筆** - 対話調の文体で執筆
4. **仕上げ** - AI 臭を排除して自然な文章に

## フェーズ判定

ユーザーのリクエストに応じて適切なフェーズから開始：

- **「記事を書いて」「note 記事作成」** → フェーズ1から順に実行
- **「読者分析」「ターゲット設定」** → フェーズ1のみ
- **「構成を考えて」「見出しを作って」** → フェーズ2のみ
- **「本文を書いて」** → フェーズ3のみ
- **「AI 臭をチェック」「自然にして」** → フェーズ4のみ

## フェーズ1: 分析

読者の悩みを具体化し、競合との差別化ポイントを見つける。

### 入力情報の収集

不明な場合はユーザーに確認：
- テーマ
- ターゲット読者
- 書く動機・自分の体験

> note というプラットフォームが何を優遇・抑制するか（一次体験優遇、煽りタイトル抑制など）は [references/note-platform.md](references/note-platform.md) を参照。差別化の方向を決める前に一読する。

### 分析プロセス

1. **読者の悩みを具体化**
   - 抽象的な悩み → 具体的なシーンに落とし込む
   - 例: 「英語学習に悩む人」→「毎日30分勉強して3ヶ月経っても海外ドラマが聞き取れない人」

2. **差別化パターンを選択**
   - 常識の反転: 「〇〇が大事」の逆を主張
   - 隠れた因果: 表面的な問題の本当の原因
   - 抽象→具体: 概念を明日からできる1アクションに
   - 比喩の発明: 複雑な概念を日常のものに例える
   - 失敗の開示: 自分の失敗談と教訓

3. **記事の価値を一文で定義**
   - 「この記事を読むと、読者は〇〇できるようになる」

### 出力形式

```markdown
## 記事分析

**ターゲット**: [具体的なペルソナ]
**悩み**: [表面的な悩み] → [本質的な悩み]
**差別化**: [採用するパターン] - [具体的な視点]
**価値**: この記事を読むと、読者は「〇〇」できるようになる
```

## フェーズ2: 構造設計

詳細は [references/structure.md](references/structure.md) と [references/note-platform.md](references/note-platform.md)（note 固有仕様）を参照。

### タイトル案（3つ作成）

パターン:
- 数字 + ベネフィット: 「〜する3つの方法」
- 意外性・逆張り: 「〜をやめた理由」
- ストーリー予告: 「〜した話をします」

ルール: **25文字程度まで**（note 公式ヘルプの推奨）、数字または意外性を含める、結論やオチまで書く。●や【】などの装飾記号は使わない

### 見出し構成

H2 は 3〜6 個。見出しだけ読んで流れがわかること。

基本フロー:
```
## 問題提起（読者の悩みに共感）
## 原因分析（なぜその問題が起きるか）
## 解決策の提示
## 具体的な手順
## まとめ
```

### リード文骨子

3〜5行で以下を含める:
1. 読者の悩みを具体的なシーンで描写
2. 自分も同じ経験をしたことを示す
3. この記事で得られる成果を示唆

> **テーマに難解な概念が含まれる場合**: `references/explain-hard-concepts.md` を参照し、リード文に「読者の現在地（既知）」を設計する（入り口の段差を小さくする）。

## フェーズ3: 本文執筆

詳細は [references/style.md](references/style.md) を参照。

### 文体の核心

「等身大の自分で、読者と対話するように書く」

`samples/` に書き手の実文がある場合は、**そこから学習した文体（フェーズ 0-b）を最優先**する。以下の基本ルールと style.md は、サンプルがない場合の既定値。

### 基本ルール

- 一人称: 「僕」または「私」（my-profile.md / samples に指定・実例があればそちらを優先）
- 語尾: 「〜なんですよね」「〜だと思うんです」
- 断定しすぎない、自己ツッコミを入れる、感情を素直に出す
- **数字を捏造しない**: 記事に書いてよい数字は、書き手の実体験の数字か、出典（URL・発行日）を確認できた一次情報のみ。「87% の企業が〜」のような雛形埋めの数字を作らない。出典不明の数字は数字なしで成立する文に書き換えるか、ユーザーに確認する

### 各セクションの流れ

```
主張（このセクションで言いたいこと）
↓
体験・具体例（なぜそう思うか）
↓
気づき・教訓（読者へのギフト）
```

### 記事の長さ目安

- ショート: 1,500〜2,000字
- ミドル: 3,000〜4,000字（デフォルト）
- ロング: 5,000字〜

## フェーズ4: AI 臭排除

詳細は [references/ai-polish.md](references/ai-polish.md) を参照。

### 禁止表現（即座に言い換え）

- 「いかがでしたか？」
- 「〜について詳しく解説します」
- 「〜を徹底解説」
- 「さあ、〜しましょう！」

### チェックポイント

1. コロン直後の箇条書きがないか
2. 「まず、次に、最後に」の機械的列挙がないか
3. 同じフレーズが3回以上繰り返されていないか
4. 抽象的な形容詞（「非常に重要」）が具体化されているか
5. 難語・専門用語で知性を見せている箇所がないか（→ 視点の転換に置き換える。マウント検査）
6. 出典・実体験の裏付けのない数字が残っていないか（捏造防止）
7. タイトルは 25 文字程度までか、煽り型テンプレになっていないか（note-platform.md）
8. my-profile.md の「書きたくないこと」「文体の好み」と矛盾していないか。samples がある場合、その文体（語尾・一人称・リズム）を再現できているか

### 修正パターン

| Before | After |
|--------|-------|
| 非常に重要です | 大事なんです / ここがポイントで |
| 〜する必要があります | 〜した方がいい |
| 様々な | いろんな |

## 完成記事の保存

保存先はユーザーに確認する。指定がなければ、カレントディレクトリに記事タイトルを反映したファイル名（例: `note下書き_タイトル.md`）で保存する。

- ユーザーの環境に記事管理の規約（Obsidian Vault のフォルダ規約など）がある場合はそれに従う
- チャット内での提示だけを求められた場合はファイル保存しない
