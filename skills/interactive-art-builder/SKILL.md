# interactive-art-builder

「体験の核（感情ゴール）」から出発し、動く・鳴る・触れる作品を完成させる。実証済みワークフロー（CatharsisField, 2026-07。参照実装: `~/Projects/visual-art`）の一般化。

## 大原則

1. **美的判断はユーザー、技術判断は AI**。体験の核・トーン・音の性格は必ず対話で確定してから実装。技術選定・実装方式は自律で進める
2. **設計正本ファースト**。実装前に architecture.md を確定し、以後の全実装（委譲含む）の契約書にする
3. **成果物は必ずユーザーが見える・聴ける形で提示**（`open` / Artifact / 公開 URL。Read のインライン表示はユーザーに見えない）
4. **数値検証**（amp・fps・パケット数）で「動いた気がする」を排除
5. 各フェーズの判断・ハマりどころを作品側 `docs/process-log.md` に記録。新規の落とし穴は本スキル `references/pitfalls.md` への追記をユーザーに提案（自己改善ループ）

## ワークフロー

### Phase 0: アイデア具体化ウィザード

`references/wizard.md` を読み、その 3 ラウンド構成で対話する。

- **AskUserQuestion が使える環境では必ずそれを使う**（1 ラウンド = 1 呼び出し、推奨に ⭐、コア操作には ASCII プレビュー、各問に「お任せ」）
- **使えない環境**（ツール一覧に無い / 呼び出しがエラー）では、wizard.md の番号付きリスト形式で同じ質問をテキスト提示し、回答（例: 「1」「2-b」）を待つ。勝手に進めない
- ユーザーが既にアイデアを語っている場合: 埋まった項目を復唱で確認し、埋まったラウンドはスキップ。差分だけ質問する
- 出力: `docs/concept.md`（コンセプトシート + 作品名候補 3 案）→ **ユーザー承認を得てから Phase 1 へ**

### Phase 1: 環境構築

選択されたプラットフォームに応じて `references/stack-web.md` または `references/stack-native.md` の環境構築節に従う。

- 導入系コマンドは公式一次情報を fetch で裏取りしてから実行（CLAUDE.md P1 ルール）
- 重いインストールはバックグラウンド実行し、待ち時間に Phase 2 を並行する

### Phase 2: 設計正本

`references/architecture-template.md` を雛形に、作品の `docs/architecture.md` を作成。

- 状態機械（正本は入力発生源のプロセス/モジュールに一元化）
- 音視覚マッピング表（操作 → 視覚反応 → 音響反応を 1 行 1 イベントで）
- プロセス間契約（ネイティブ: OSC アドレス・ポート・引数型の表 / ウェブ: AudioEngine インターフェース）
- **Tier 設計**: Tier 1（コア体験、単独で成立）と Tier 2（パターン層等のオプショナル強化）を分離し、依存の重い層を非必須化する

### Phase 3: 実装（並行委譲）

architecture.md を契約書として渡し、並行実装する。

- **視覚（粒子・シェーダ・演出）→ sonnet サブエージェント**
- **音響（シンセエンジン）→ opus サブエージェント**
- **パターン層（Tidal / Strudel）→ メインセッションで直接書く**（委譲しない）
- 委譲プロンプトには該当スタックの `references/` から**既知の落とし穴を明記**する（例: 「colorMode のレンジと alpha 値を突き合わせること」）。これが手戻りを最も減らす
- 委譲成果物はメインが必ずレビュー + 実測検証してから統合

### Phase 4: 統合・自動検証

`references/verification.md` のハーネスで、ユーザーに触らせる前に自動で裏取りする。

- 起動スクリプト（テンプレあり）で全プロセス起動 → **ログが正常でも pgrep で生存確認**
- E2E: 合成イベント（PointerEvent / OSC 注入）で全インタラクションを発火
- 数値: 音響 amp サンプリング、fps とフレーム間隔分散（30fps 固定 + 分散小 = rAF 制限であり負荷ではない）
- 音出しテストの前にシステム音量を退避して 40〜45% に絞り、終了後復元する

### Phase 5: 体感チューニング

- 成果物を提示（ネイティブ: 起動して操作を案内 / ウェブ: URL）し、体感フィードバックを待つ
- 全チューニング定数はファイル冒頭に一元化し、README にノブ一覧表を置く（フィードバック → 即調整のループを短くする）

### Phase 6: 公開（ユーザーが希望した場合のみ）

- ウェブ: `wrangler deploy`（静的アセット）→ 本番 URL で E2E 再実行 → og:image は絶対 URL で再デプロイ
- **ライセンス確認**: 依存に AGPL 系（Strudel 等）があれば「ソース公開が必要」を必ず告知し、方針の確認を取ってから公開
- OG 画像は `canvas.toDataURL()` で決定的瞬間を原子的にキャプチャ（MCP スクショはレイテンシで瞬間を外す）

### Phase 7: 記録と還流

- `docs/process-log.md` を完成させ、コミット提案
- 今回新たに踏んだ落とし穴があれば、`references/pitfalls.md` への追記をユーザーに提案

## references の読み方（progressive disclosure）

| ファイル | 読むタイミング |
|---|---|
| `references/wizard.md` | Phase 0 開始時（必須） |
| `references/architecture-template.md` | Phase 2 |
| `references/stack-web.md` / `stack-native.md` | Phase 1・3（選択したスタックのみ） |
| `references/verification.md` | Phase 4・6 |
| `references/pitfalls.md` | Phase 3 の委譲プロンプト作成時 + トラブル時 |
| `templates/` | Phase 3（骨格として作品にコピーして肉付け） |

フルコードの参照実装: `~/Projects/visual-art`（ネイティブ + ウェブ両方。存在しない場合は templates のみで進める）。
